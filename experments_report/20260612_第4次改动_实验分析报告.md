## 一、模型架构总览 (V2.4)

### 1.1 整体结构

```
DualBranchSTImputer
├── MultiScaleMoEBackbone (Main Branch)
│   ├── ScaleTokenEncoder ×3  → h_f, h_m, h_c
│   ├── QualityRouter ×3      → gate_f, gate_m, gate_c
│   ├── TopKRoutedExpertPool  → z_f, z_m, z_c (共享参数)
│   ├── ExpertEnhancedSharedInput  → h_f_shared, h_m_shared, h_c_shared  [NEW V2.4]
│   ├── GatedCrossScaleSharedExpert → z_shared (含 ReliabilityAwareScaleGate)
│   ├── ProgressiveRouteFusion → h_route (scale_mode-aware)
│   ├── SharedRoutedResidualFusion → h_main (含 AdaptiveBranchGate)  [ENHANCED V2.4]
│   └── PredHead → x_hat_main
└── NullResidualBranch (Aux Branch, 当前 disabled)
```

### 1.2 V2.4 新增/增强组件

#### A. ExpertEnhancedSharedInput (`fusion.py:175-216`)

让 Shared Branch 可以使用 post-expert 特征，而非只能使用 pre-expert embedding。

| 模式 | 公式 | 说明 |
|------|------|------|
| `pre` | `h_s` | 原始 embedding（与 V2.3 相同） |
| `post` | `z_s` | 纯 expert 特征 |
| `hybrid` | `h_s + β_s · z_s` | **推荐模式**，可学习 β ∈ (0,1) |
| `concat_hz` | `Conv1x1([h_s, z_s])` | 拼接后投影 |

```python
# β 通过 sigmoid 约束到 (0,1)，初始 logit(0.1) → β≈0.1
self.beta = nn.Parameter(torch.full((3,), torch.logit(beta_init).item()))
```

#### B. AdaptiveBranchGate (`fusion.py:219-260`)

样本级 MLP 门控，输入 `[pool(h_shared) || pool(h_route) || q_f || scale_gate]`，输出 `[B,2]` softmax 权重。

| init_mode | 初始偏置 | softmax 初始值 | 适用场景 |
|-----------|----------|---------------|---------|
| `balanced` | zeros | [0.5, 0.5] | 通用 |
| `shared_bias` | [1, -1] | [0.88, 0.12] | BikeNYC（shared 主导） |
| `route_bias` | [-1, 1] | [0.12, 0.88] | TaxiBJ（预期 route 主导） |

#### C. SharedRoutedResidualFusion 三种模式 (`fusion.py:376-442`)

| 模式 | 公式 | 说明 |
|------|------|------|
| `residual` | `h_main = h_shared + σ(γ) · h_route_proj` | V2.3 旧逻辑 |
| `routed_primary` | `h_main = h_route_proj + σ(γ) · h_shared` | 以 route 为主 |
| `adaptive_gate` | `h_main = w_s·h_shared + w_r·h_route_proj` | **推荐**，样本级自适应 |

### 1.3 完整前向流程

```
x_f, m_f / x_m, m_m / x_c, m_c
        │
        ▼
ScaleTokenEncoder ×3  ──→  h_f, h_m, h_c
        │
        ▼
QualityRouter ×3  ──→  gate_f, gate_m, gate_c
        │
        ▼
TopKRoutedExpertPool (共享)  ──→  z_f, z_m, z_c
        │
        ├──[Routed Branch]──→ ProgressiveRouteFusion(scale_mode)
        │                            │
        │                     fine_mid: z_m→upsample→GatedFusion2(z_f, z_m↑)
        │                     fine_mid_coarse: z_c→up→GatedFusion2(z_m, z_c↑)
        │                                      →up→GatedFusion2(z_f, z_mc↑)
        │                            │
        │                            ▼
        │                         h_route → route_proj → h_route_proj
        │
        ├──[Shared Branch]──→ ExpertEnhancedSharedInput(h, z)
        │                            │
        │                     hybrid: h_s + β_s · z_s
        │                            │
        │                            ▼
        │                     ReliabilityAwareScaleGate(q, r, active_mask)
        │                            │
        │                     MLP([pool(h)||q||r]) → softmax → [w_f, w_m, w_c]
        │                            │
        │                            ▼
        │                     GatedCrossScaleSharedExpert
        │                     weighted_fuse → shared_refine → h_shared
        │
        └──[Fusion]──→ AdaptiveBranchGate(h_shared, h_route_proj, q_f, scale_gate)
                            │
                     MLP → softmax → [w_shared, w_route]
                            │
                            ▼
                     h_main = w_shared·h_shared + w_route·h_route_proj
                            │
                            ▼
                     PredHead → x_hat_main
```

### 1.4 损失函数

```text
L = L_main(masked SmoothL1)
  + λ_cross · warmup(t) · L_cross_scale          # fine→mid (→coarse)
  + λ_importance · warmup(t) · L_importance       # 专家使用均衡
  + λ_load · warmup(t) · L_load                   # 专家负载均衡
  + λ_fusion_entropy · L_fusion_entropy           # 默认 0
  + λ_branch_entropy · L_branch_entropy           # 默认 0

warmup(t) = min(1.0, epoch / aux_loss_warmup_epochs)
```

|:---:|---|:---:|:---:|:---:|:---:|:---:|:---:|
| 🥇 | Fine-Only | 84 | **12.8688** | 21.28 | 12.1452 | +6.0% ❌ | 100 |

> ⚠️ **所有模型全面退化**。此前 V2.3 的最佳模型 Routed-Only (11.90) 退化为最差模型 (20.80)。

### 2.2 退化原因分析

#### 根因 1: 超参数全面变更

V2.3 → V2.4 的 config 变更（影响所有模型，包括不使用新组件的消融实验）：

| 参数 | V2.3 | V2.4 | 影响 |
|------|------|------|------|
| `lr_main` | **0.001** | 0.0008 | -20% 基础学习率 |
| `scheduler` | cosine | **warmup_cosine** (5ep) | 前5轮 LR 线性爬升 |
| `epoch 1 有效 LR` | 0.001 | **0.00016** | **6.25× 更小** |
| `lambda_cross` | 0.03 | 0.01 | -67% 跨尺度约束 |
| `lambda_importance` | 0.001 | 0.0005 | -50% |
| `lambda_load` | 0.001 | 0.0005 | -50% |
| `dropout` | 0.1 | 0.05 | 减半 |
| `early_stopping` | ❌ 无 | patience=20 | 提前终止 |
| `aux_loss_warmup` | ❌ 无 | 10 epochs | 辅助 loss 慢启动 |

#### 根因 2: `route_bias` 初始化不当

TaxiBJ 配置了 `branch_gate_init: "route_bias"`（初始 softmax ≈ [0.12 shared, 0.88 route]），但实际训练结果：

```
Full Model:
  branch_gate_shared_mean = 1.000000
  branch_gate_route_mean  = 0.000000 (1.38e-09)
```

**门控完全坍塌到 shared=1.0**，与预期完全相反。模型在早期 epoch 需要消耗大量梯度来克服 route_bias，导致训练效率极低。

#### 根因 3: Early Stopping 过早终止

所有模型均提前终止（46-93 epoch），而 V2.3 均跑到 100-120 epoch：

| 模型 | V2.4 终止 Ep | V2.3 终止 Ep |
|------|:---:|:---:|
| Full | 68 | 120 |
| Routed-Only | 58 | 100 |
| Shared-Only | 74 | 100 |
| Fixed Experts | 46 | 100 |

低 LR + warmup 导致收敛速度变慢，early stopping 在模型仍在缓慢改进时就终止了训练。

#### 根因 4: 初始化随机状态偏移

V2.4 新增了 `ExpertEnhancedSharedInput`、`AdaptiveBranchGate` 等模块，即使消融实验不使用它们，这些模块的初始化也会消耗 PyTorch 全局随机数，导致相同 seed=42 下已有模块（Embedding、Expert、Router）的初始权重不同。V2.4 Routed-Only 的第一轮 MAE 为 64.71，而 V2.3 为 45.36（+43%），证实了这一点。

### 2.3 关键诊断指标

#### Full Model (最后一轮 ep68)

```text
scale_gate:        f=0.675  m=0.325  c=0.000  (fine_mid, coarse masked)
branch_gate:       shared=1.000  route≈0      (完全坍塌到 shared)
shared_input_beta: f=0.308  m=0.599  c=0.223  (mid 尺度 expert 特征融合最多)
effective_route_ratio: 0.850                   (h_route_proj 的 norm 仍存在但不参与融合)
route_gamma:       0.119                        (residual 路径的 gamma，adaptive_gate 模式下不使用)
```

**解读**: `effective_route_ratio=0.85` 说明 h_route_proj 的幅度是 h_shared 的 85%，但由于 branch_gate 将 route 权重压到接近 0，这些特征完全被浪费。

#### Shared-Only (最后一轮 ep74)

```text
scale_gate:        f=0.695  m=0.305  c=0.000
shared_input_beta: f=0.301  m=0.379  c=0.237
```

Shared-Only 的 beta 值与 Full Model 接近，但 MAE 好于 Full (19.61 vs 18.81 最佳)，说明在 V2.4 配置下，route branch 的加入反而损害了性能。

#### Fine-Only — 唯一幸存的模型

Fine-Only 仅退化 6%（12.15→12.87），原因：
- 不使用 shared branch → ExpertEnhancedSharedInput 不参与
- 不使用 multiscale → cross_scale_loss=0、scale_gate 固定 [1,0,0]
- 不使用 branch fusion → AdaptiveBranchGate 不参与
- 受影响的仅有 LR、scheduler、dropout 和 balance loss 权重

6% 的退化可归因于 LR 降低 + warmup + 初始化随机偏移。

## 四、架构洞察

### 4.1 ExpertEnhancedSharedInput 的 beta 学习

两个数据集上，`hybrid` 模式的 beta 值呈现一致规律：

| 数据集 | β_f | β_m | β_c | 模式 |
|--------|:---:|:---:|:---:|------|
| TaxiBJ Full | 0.31 | **0.60** | 0.22 | mid 尺度最需要 expert 增强 |
| TaxiBJ Shared-Only | 0.30 | 0.38 | 0.24 | 较均衡 |
| BikeNYC Full | 0.16 | 0.21 | **0.23** | 粗尺度最需要 expert 增强 |
| BikeNYC Shared-Only | 0.13 | 0.20 | 0.22 | 一致偏低 |

**规律**: 数据更难的尺度需要更多 expert 特征注入。TaxiBJ 的 mid 尺度（32→16）信息损失大，β_m=0.60 表明需要大量 expert 补偿。

### 4.2 AdaptiveBranchGate 的实际行为

两个数据集上，adaptive gate 都没有实现「自适应」的预期——它坍缩到了一个极端：

- **TaxiBJ**: 坍缩到 shared=1.0（尽管 route_bias 初始化！）
- **BikeNYC**: 收敛到 shared=0.975（与 shared_bias 初始化一致）

**结论**: 当前的数据和任务不足以让 sample-level branch gating 展现出有意义的动态范围。所有样本的 branch gate 几乎相同，MLP 学到的实际上是一个「全局偏好」而非「样本级自适应」。

### 4.3 Route Branch 的实际贡献

```
effective_route_ratio = ||h_route_proj|| / ||h_shared||

TaxiBJ Full:  0.85  (route 特征幅度大，但 gate 将其清零！)
BikeNYC Full: 0.74  (route 特征幅度中等，gate 将其压到 2.5%)
```

`effective_route_ratio` 表明 route branch 确实产生了有意义的特征（幅度与 shared 可比），但 adaptive gate 选择忽略它们。这可能意味着：
1. Route 特征与 shared 特征高度冗余
2. Route 特征包含噪声，直接加入会损害性能
3. Adaptive gate 退化到保守策略（完全信任 shared）

## 五、问题与建议

### 5.1 TaxiBJ 实验失败的根本原因

**不是架构问题，而是超参数配置问题**：

1. **`lr_main=0.0008` + `warmup_cosine`** → epoch 1 有效 LR 仅 0.00016，是 V2.3 的 1/6.25
2. **`branch_gate_init=route_bias`** → 模型需要消耗早期 epoch 对抗初始化偏置
3. **`early_stopping`** → 在模型因低 LR 而缓慢改进时就终止了
4. **`aux_loss_warmup=10`** → 辅助 loss 在前 10 轮被严重压低

### 5.2 建议的修复方案

#### 方案 A: 回滚超参数 + 保留架构（推荐）

```json
// TaxiBJ 建议配置
{
  "lr_main": 0.001,                    // 回滚到 V2.3 值
  "scheduler": { "type": "cosine" },   // 去掉 warmup
  "branch_gate_init": "shared_bias",   // TaxiBJ 实际也是 shared 主导！
  "lambda_cross": 0.03,                // 回滚
  "lambda_importance_balance": 0.001,  // 回滚
  "lambda_load_balance": 0.001,        // 回滚
  "dropout": 0.1,                      // 回滚
  "early_stopping": { "enabled": false } // 关闭或增加 patience 到 40
}
```

这样可以在保留 V2.4 架构改进的同时，恢复 V2.3 的训练稳定性。

#### 方案 B: 仅验证架构改进

运行以下消融实验来隔离超参数影响：
1. V2.4 架构 + V2.3 超参数 → 验证架构是否有效
2. V2.3 架构 + V2.4 超参数 → 验证超参数是否是罪魁祸首
3. `branch_gate_init=shared_bias` vs `route_bias` → 验证初始化偏置的影响

#### 方案 C: Fix TaxiBJ epochs

当前 TaxiBJ config 中有 `epochs: 120` 但被 ablation override 为 `100`。建议统一为 120 并关闭 early stopping，给模型足够的收敛时间。

### 5.3 BikeNYC 建议

BikeNYC 表现稳定，可以：
1. 将 `epochs` 从 100 降低到 **80**（最佳通常在 ep50-70）
2. 考虑将 `branch_fusion_mode` 回退到 `residual`（因为 adaptive_gate 坍缩到 shared=1.0，与 residual+小 gamma 等价但多了不必要的参数）

## 六、总结

| 维度 | TaxiBJ | BikeNYC |
|------|--------|---------|
| V2.4 架构 | ✅ 代码正确，逻辑无误 | ✅ 代码正确，逻辑无误 |
| 超参数 | ❌ 严重倒退 (LR↓, warmup, route_bias) | ✅ 基本合理 (shared_bias) |
| 实验结果 | ❌ 全面退化 32-75% | ✅ 基本持平 ±2% |
| 门控行为 | 坍缩到 shared=1.0 | 收敛到 shared=0.975 |
| ExpertEnhancedShared | β 学习有意义 (mid=0.60) | β 学习有意义 (coarse=0.23) |
| AdaptiveBranchGate | 无样本级动态范围 | 无样本级动态范围 |

**核心结论**: V2.4 的架构改进（ExpertEnhancedSharedInput、AdaptiveBranchGate、三种 fusion 模式）代码实现正确，但 TaxiBJ 的**超参数配置需要修正**。BikeNYC 结果证实了架构的稳定性。AdaptiveBranchGate 在当前的样本级别上没有展现出有意义的自适应能力，可能需要在更细粒度（patch 级）或更强的条件信号下才能发挥作用。
