## 1. 模型架构（第5次改动 - V2.5 当前版本）

### 1.1 整体流程：MultiScaleMoEBackbone

模型前向传播分为 7 个阶段（[main_branch.py](src/stmoe_imputer/models/main_branch.py):204-432）：

```
Step 1: 多尺度嵌入          → h_f, h_m, h_c
Step 2: 质量路由             → gate_f, gate_m, gate_c
Step 3: 专家计算 (条件)      → z_f, z_m, z_c + top_k/weights
Step 4: 共享分支             → h_f_shared(输入适配) → z_shared(跨尺度融合)
Step 5: 路由分支             → h_route(渐进式尺度融合)
Step 6: 分支融合             → h_main(共享+路由融合)
Step 7: 预测                 → x_hat_main (+ x_hat_shared, x_hat_route aux)
```

### 1.2 组件详解

#### Step 1: ScaleTokenEncoder（[embedding.py](src/stmoe_imputer/models/embedding.py)）

三尺度独立嵌入，每个尺度包含：
- `value_embed`: Conv3d(c_in→dim) 将原始输入映射到隐空间
- `mask_embed`: Conv3d(1→dim) 缺失指示
- `scale_embed`: 可学习尺度标记向量
- `time_embed` + `space_embed`: 时空位置编码

```python
h_f = embed_f(x_f, m_f)  # fine:   [B, 64, 12, 32, 32]
h_m = embed_m(x_m, m_m)  # mid:    [B, 64, 12, 16, 16]
h_c = embed_c(x_c, m_c)  # coarse: [B, 64, 12, 8, 8]
```

#### Step 2: QualityRouter（[router.py](src/stmoe_imputer/models/router.py):7-24）

样本级质量感知路由器：

```
router(h) = softmax(Linear(GELU(Linear([pool(h) || obs_stats || scale_embed]))))
```

- 输入: 池化特征(dim) + 观测统计(q_dim=5) + 尺度嵌入(dim)
- `obs_stats`: missing_rate, observed_ratio, temporal_missing_score, spatial_missing_score, aggregation_reliability

#### Step 3: TopKRoutedExpertPool（[experts.py](src/stmoe_imputer/models/experts.py):25-79）

- `num_experts=4`, `top_k=2`, 支持 `topk`/`dense`/`soft_topk` 三种模式
- 共享专家池（`share_experts=True`）：mid/coarse 共享 fine 的专家参数
- STExpert: Conv3d → GroupNorm → GELU → ResidualSTBlock
- **条件计算逻辑**（V2.5 关键）：

```python
need_expert_features = use_routed_branch or (
    use_shared_branch and shared_input_mode != "pre"
)
# V2.5: shared_input_mode="pre" → need_expert_features = use_routed_branch
# Shared-Only 模式跳过专家计算，z_f/m/c 全为零
```

#### Step 4a: ExpertEnhancedSharedInput（[fusion.py](src/stmoe_imputer/models/fusion.py):175-216）

**V2.5 使用 `mode="pre"`**——这是关键修复：

| 模式 | 公式 | 行为 |
|---|---|---|
| `pre` | `return h_f, h_m, h_c` | 纯嵌入，不注入专家特征 ✅ |
| `post` | `return z_f, z_m, z_c` | 纯专家输出 |
| `hybrid` | `return h + σ(β)·z` | 嵌入+专家混合 ❌ (V2.4 根因) |
| `concat_hz` | `Conv1x1([h, z])` | 通道拼接投影 |

在 `pre` 模式下，专家特征完全不进入共享分支，共享分支直接接收原始嵌入。

#### Step 4b: GatedCrossScaleSharedExpert（[fusion.py](src/stmoe_imputer/models/fusion.py):109-172）

跨尺度共享专家融合：

```
z_shared = Conv1d( [w_f·h_f || w_m·h_m_up || w_c·h_c_up] )
          → ResidualSTBlock × 2
```

- **ReliabilityAwareScaleGate**（[fusion.py](src/stmoe_imputer/models/fusion.py):55-106）:

```
scale_weight = softmax(MLP([pool(h_f)||pool(h_m)||pool(h_c)||q_f||q_m||q_c||r_m_mean||r_c_mean]))
```

- MLP: 384 → 128 → 3（fine/mid/coarse 维度）→ softmax
- `r_m`/`r_c` 为可选可靠性输入（aggregation_reliability）
- `active_mask`: inactive 尺度被mask（scale_mode="fine_mid" 时 coarse 被屏蔽）

#### Step 5: ProgressiveRouteFusion（[fusion.py](src/stmoe_imputer/models/fusion.py):296-373）

scale_mode 感知的路由分支融合：

| scale_mode | 路由融合路径 |
|---|---|
| `fine` | `h_route = ResidualSTBlock(z_f)` |
| `fine_mid` | `z_m→upsample→GatedFusion2(z_f, z_m_up)` |
| `fine_mid_coarse` | `z_c→upsample→GatedFusion2(z_m, z_c_up)→upsample→GatedFusion2(z_f, z_mc_up)` |

GatedFusion2: `softmax(Conv1x1([x1||x2])) · [x1,x2] → ResidualSTBlock`

#### Step 6: SharedRoutedResidualFusion（[fusion.py](src/stmoe_imputer/models/fusion.py):376-445）

**V2.5 使用 `branch_fusion_mode="residual"`**——这是第二个关键修复：

```python
# residual 模式
gamma = sigmoid(self.route_gamma)       # 标量可学习门控
h_main = h_shared + gamma * h_route_proj # 线性组合
```

与 V2.4 的 `adaptive_gate` 模式对比：

| 模式 | 融合公式 | 适应性 |
|---|---|---|
| `residual` (V2.5) | `h_shared + σ(γ)·h_route_proj` | 标量门控，全局共享 |
| `routed_primary` | `h_route_proj + σ(γ)·h_shared` | 标量门控，全局共享 |
| `adaptive_gate` (V2.4) | `w_s·h_shared + w_r·h_route_proj` | MLP 样本级门控 ❌ |

#### Step 7: 预测头

- 主预测头: Conv3d→GELU→Conv3d (dim→dim/2→c_in)
- 辅助头（仅 full 模式）: shared_aux_head, route_aux_head（结构相同）

### 1.3 损失函数（[losses.py](src/stmoe_imputer/losses.py)）

```
L_total = L_main(masked SmoothL1)
        + λ_cross · L_cross_scale           # 跨尺度一致性
        + λ_importance · L_importance_balance # 专家使用均衡
        + λ_load · L_load_balance           # 专家负载均衡
        + λ_complementary · L_complementary  # 分支互补 (full模式)
        + λ_shared_aux · L_shared_aux       # 共享辅助
        + λ_route_aux · L_route_aux         # 路由辅助
        + λ_fusion_entropy · L_fusion_entropy  # (未启用)
        + λ_branch_entropy · L_branch_entropy  # (未启用)
```

aux_loss_warmup: 线性预热因子 `min(1, epoch/warmup_epochs)` 作用于 cross_scale 和 balance 损失。

## 2. 关键改动：V2.4-rollback → V2.5

从回滚版本（第4次改动回滚后）到第5次改动，TaxiBJ 只修改了 5 个参数：

| 参数 | V2.4-rollback | V2.5（当前） | 修改原因 |
|---|---|---|---|
| `shared_input_mode` | `"hybrid"` | **`"pre"`** | 核心修复：阻止专家特征进入共享分支 |
| `branch_fusion_mode` | `"adaptive_gate"` | **`"residual"`** | 回退到标量门控，避免样本级gate坍缩 |
| `route_gamma_init` | -4.0 | **-3.0** | 微调：给 routed branch 略高的初始贡献 |
| `gate_lr_mult` | 2.0 | **1.0** | 恢复正常门控学习率 |
| `scalar_lr_mult` | 5.0 | **2.0** | 恢复正常标量参数学习率 |

**未修改的 V2.4 特性**（经过验证是安全的）：
- `use_scale_gate=true` + `use_reliability_gate=true` — ReliabilityAwareScaleGate 保留
- `enable_branch_aux=true` + `enable_complementary_loss=true`
- `scale_mode="fine_mid"`
- 其他 V2.4 架构组件（ProgressiveRouteFusion, QualityRouter, TopKRoutedExpertPool）

## 5. 根因分析：ExpertEnhancedSharedInput 为什么破坏 TaxiBJ？

### 5.1 实验验证

通过对照实验可以 100% 确定根因：

| 实验条件 | 是否使用 ExpertEnhancedSharedInput | MAE |
|---|---|---|
| V2.4-rb Shared-Only (hybrid) | ✅ h + β·z | 19.00 |
| V2.5 Shared-Only (pre) | ❌ h only | 12.27 |
| V2.4-rb Full (hybrid + adaptive_gate) | ✅ | 19.44 |
| V2.5 Full (pre + residual) | ❌ (pre) | 11.80 |

Shared-Only 是 "最小对照" 实验——它只使用共享分支，不涉及 routed branch 和 branch fusion。Shared-Only 在 hybrid→pre 切换后 MAE 从 19.00 降至 12.27（-35.4%），证明 `ExpertEnhancedSharedInput(hybrid)` 本身就是退化源。

### 5.2 机制猜想

```
TaxiBJ 特征空间特点：
- 时空模式简单但局部性强（城市交通流）
- 嵌入特征 h 编码了完整的局部上下文
- 专家特征 z 引入了全局信息（通过 top-k 稀疏激活）

hybrid 模式: h + β·z（β=0.1 初始）
- 即使是微小的 β=0.1，也在共享分支输入中混入了 ~10% 的专家特征
- 共享分支的 GatedCrossScaleSharedExpert 对输入质量高度敏感
- β 虽然可学习（logit_beta 参数），但在 V2.4-rb 中 β 被冻结在 0.1（loss 梯度未推动 β 变化）

BikeNYC 为什么不受影响：
- 时空模式更复杂（共享单车出行），特征维度更高
- expert 特征与 embedding 特征方向更一致
- β·z 是增量贡献而非噪声
- 或者 scale_mode=fine_mid_coarse 的额外 coarse 信息缓冲了影响
```

### 5.3 AdaptiveBranchGate 坍缩问题

在 V2.4 和 V2.4-rb 中，AdaptiveBranchGate 在 TaxiBJ 上坍缩到 `shared=1.0, route=0.0`（std=0，完全无样本级适应性）。在 V2.5 的 `residual` 模式下，AdaptiveBranchGate 不再被使用——改用标量 `route_gamma` 进行融合。因此，AdaptiveBranchGate 设计本身可能对 TaxiBJ 这样的数据集不适用。

## 6. 消融实验分析

### 6.1 TaxiBJ 消融排序（V2.5）

```
Full             11.80  ← 最优，两分支互补
No-Cross-Scale   11.88  ← 无共享分支，纯路由融合
Routed-Only      11.88  ← 纯路由分支
Fine-Only        12.13  ← 纯 fine 尺度
No-Router        12.25  ← 无学习路由（dense路由）
Shared-Only      12.27  ← 纯共享分支
Fixed-Scale-Exp  17.44  ← 无路由 + 无共享专家 + 无共享分支
```

### 6.2 解释

1. **Full > 单分支**：11.80 < min(11.88, 12.27)，两分支互补有效但增益很小（-0.6% vs Routed-Only），说明 routed branch 已经接近最优
2. **Routed-Only ≈ No-Cross-Scale**：两者都是 routed-only 路径（无 shared branch），相同的有效路径，MAE 一致（11.88），验证了实验的一致性
3. **Shared-Only < Routed-Only**：12.27 > 11.88，纯嵌入共享分支比纯路由专家分支差约 3.3%
4. **Fine-Only 接近 Full**：12.13 vs 11.80（差 2.8%），多尺度带来的收益有限但存在
5. **Fixed-Scale-Experts 垫底**：缺少路由器 + 缺少跨尺度共享专家 + 无共享专家参数 = 17.44

### 6.3 BikeNYC 消融排序

```
Full             2.82  ← 最优
Shared-Only      2.84  ← 非常接近 Full
Routed-Only      2.89
No-Cross-Scale   2.90
Fixed-Scale-Exp  2.98
Fine-Only        3.01
No-Router        3.01
```

BikeNYC 所有消融 MAE 在 2.82-3.01 范围内（差异 <7%），模型对架构变化不敏感。Shared-Only (2.84) 接近 Full (2.82)，说明跨尺度共享专家已经提取了大部分有效信息。

## 7. 结论与建议

### 7.1 核心结论

1. **ExpertEnhancedSharedInput(hybrid) 是 TaxiBJ 退化的唯一架构根因**——切换到 `"pre"` 模式后所有退化模型完全恢复
2. **AdaptiveBranchGate 对 TaxiBJ 不适用**——会坍缩为全局常量，`"residual"` 模式（标量门控）更安全
3. **V2.5 是 TaxiBJ 的历史最佳版本**：Full MAE=11.80（vs V2.3 最佳 12.01）
4. **BikeNYC 全版本稳定**，架构变化的影响在噪声范围内

### 7.2 当前配置评估

**TaxiBJ 当前配置是目前最优的保守配置：**
- `shared_input_mode="pre"` ✅ 稳定
- `branch_fusion_mode="residual"` ✅ 稳定
- `route_gamma_init=-3.0` ✅ 适中（γ 收敛到 0.137）
- 保留的 V2.4 特性（ReliabilityAwareScaleGate, ProgressiveRouteFusion）✅ 安全

**BikeNYC 当前配置无需修改。**

### 7.3 后续方向

1. **探索 Hybrid 模式的正确使用方式**：
   - 调低 `shared_expert_beta_init`（0.01 代替 0.1）可能避免 "污染"
   - 或对 β 使用独立学习率组（如 lr×10）
   - 仅对特定尺度（如 coarse）启用 hybrid

2. **探索 AdaptiveBranchGate 适用条件**：
   - 可能只在 scale_mode=fine_mid_coarse（有 coarse 信息）时才有效
   - 可尝试添加 gate 熵正则化（`lambda_branch_entropy > 0`）防止坍缩

3. **TaxiBJ 路由贡献提升**：
   - 当前 `route_gamma→0.137`，路由分支贡献仅 ~14%
   - 可尝试 `route_gamma_init=-2.0`（初始 σ(-2)=0.12）让路由分支更早参与

4. **Fixed-Scale-Experts 消融修复**：该消融的退化源自 `use_router=False + share_experts=False + use_shared_branch=False` 三重禁用。如需测试 "固定尺度门控"，应保持 `use_shared_branch=True`，仅设置 `use_scale_gate=False`。
