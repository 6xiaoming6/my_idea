## 一、模型架构总览 (V2.4)

### 1.1 顶层结构

```
DualBranchSTImputer
├── MultiScaleMoEBackbone (Main Branch)           # 核心计算图
│   ├── [Embedding] ScaleTokenEncoder ×3          # h_f, h_m, h_c
│   ├── [Router] QualityRouter ×3                 # gate_f, gate_m, gate_c
│   ├── [Expert] TopKRoutedExpertPool (共享)       # z_f, z_m, z_c
│   ├── [NEW] ExpertEnhancedSharedInput           # h_f_shared, h_m_shared, h_c_shared
│   ├── GatedCrossScaleSharedExpert               # z_shared (含 ReliabilityAwareScaleGate)
│   ├── ProgressiveRouteFusion                    # h_route (scale_mode-aware)
│   ├── [ENH] SharedRoutedResidualFusion          # h_main (含 AdaptiveBranchGate)
│   └── PredHead                                  # Conv3d → GELU → Conv3d
└── NullResidualBranch (Aux, 当前 disabled)
```

### 1.2 详细前向流程

```
Step 1: Embedding
  x_f,m_f / x_m,m_m / x_c,m_c
    → ScaleTokenEncoder(value_embed + mask_embed + scale_embed + time_embed + space_embed)
    → h_f, h_m, h_c                                                          [B,64,T,H,W]

Step 2: Routing
  QualityRouter(pool(h) || obs_stats(m) || scale_embed_vec)
    → softmax → gate_f, gate_m, gate_c                                       [B,4]

Step 3: Expert Computation (条件执行)
  need_expert_features = use_routed_branch ∨ (use_shared_branch ∧ shared_input_mode≠"pre")
  TopKRoutedExpertPool(h, gate, top_k=2)
    → z_f, z_m, z_c                                                          [B,64,T,H,W]
    → top_indices, top_weights, selected_mask

Step 4: Shared Branch (if use_shared_branch)
  ExpertEnhancedSharedInput(h_f,h_m,h_c, z_f,z_m,z_c)
    mode=hybrid: h_s_shared = h_s + β_s · z_s    (β ∈ [0,1], 可学习)
    mode=pre:     h_s_shared = h_s                (V2.3 等价)
    mode=post:    h_s_shared = z_s
    mode=concat_hz: h_s_shared = Conv1x1([h_s, z_s])
    → h_f_shared, h_m_shared, h_c_shared

  ReliabilityAwareScaleGate(pool(h) || q || r_mean, active_mask)
    → MLP(384→128→3) → softmax → scale_weight [w_f, w_m, w_c]               [B,3]
    (active_mask 将非活跃尺度 logit 置为 -inf)

  GatedCrossScaleSharedExpert
    h_m↑, h_c↑ → concat([w_f·h_f, w_m·h_m↑, w_c·h_c↑])
    → Conv1d(192→64) → 2×ResidualSTBlock
    → z_shared                                                               [B,64,T,H,W]
    → shared_refine (2×ResidualSTBlock)
    → h_shared

Step 5: Route Branch (if use_routed_branch)
  ProgressiveRouteFusion(z_f, z_m, z_c, scale_mode)
    scale_mode=fine:           h_route = fine_refine(z_f)
    scale_mode=fine_mid:       z_m↑ → GatedFusion2(z_f, z_m↑) → h_route
    scale_mode=fine_mid_coarse: z_c↑→GatedFusion2(z_m,z_c↑)→z_mc↑→GatedFusion2(z_f,z_mc↑)
    → h_route
    → route_proj (Conv1d + ResidualSTBlock)
    → h_route_proj

Step 6: Branch Fusion
  SharedRoutedResidualFusion(h_shared, h_route_proj, q_f, scale_gate)
    mode=residual:        h_main = h_shared + σ(γ_r) · h_route_proj
    mode=routed_primary:  h_main = h_route_proj + σ(γ_s) · h_shared
    mode=adaptive_gate:   AdaptiveBranchGate(pool(h_shared)||pool(h_route)||q_f||scale_gate)
                          → MLP(136→128→2) → softmax → [w_shared, w_route]
                          h_main = w_shared·h_shared + w_route·h_route_proj

Step 7: Prediction
  PredHead: Conv3d(64→32) → GELU → Conv3d(32→c_in) → x_hat_main
```

### 1.3 损失函数

```
L = L_main(masked SmoothL1, 仅计算缺失位置)
  + λ_cross · warmup(t) · L_cross_scale          # 预测→mid尺度一致性
  + λ_importance · warmup(t) · L_importance       # 专家重要性均衡
  + λ_load · warmup(t) · L_load                   # 专家负载均衡
  + λ_fusion_entropy · L_fusion_entropy           # 融合门控熵 (默认0)
  + λ_branch_entropy · L_branch_entropy           # 分支门控熵 (默认0)

warmup(t) = min(1.0, epoch / aux_loss_warmup_epochs)
```

### 2.3 关键发现

#### 发现 1: 不使用 V2.4 新组件的模型恢复良好

Fine-Only、No Cross-Scale、Routed-Only 三个模型不使用 ExpertEnhancedSharedInput 和 AdaptiveBranchGate，回滚后大幅改善：

- **Fine-Only**: 完全恢复到 V2.3 水平（12.04 vs 12.15），验证了超参数回滚的有效性
- **Routed-Only**: 从 20.80 改善到 13.28，但仍比 V2.3（11.90）差 11.6%
- **No Cross-Scale**: 从 19.53 改善到 13.12，但仍比 V2.3（12.03）差 9.1%

Routed-Only 和 No Cross-Scale 未完全恢复的可能原因：
1. 初始化随机状态偏移（V2.4 新增模块消耗了随机数）
2. `gate_lr_mult=2.0` 改变了门控参数的学习动态

#### 发现 2: 使用 ExpertEnhancedSharedInput 的模型持续恶化

Shared-Only（MAE 19.00 vs V2.3 的 14.82，+28%）和 Full（MAE 19.44 vs V2.3 的 12.01，+62%）即使回滚后仍远差于 V2.3。**这两个模型的唯一共同架构变化是 ExpertEnhancedSharedInput**。

Shared-Only 的前向路径:
```
V2.3: h_f,m,c → CrossScaleSharedExpert → z_shared → refine → h_shared → PredHead
V2.4: h_f,m,c → ExpertEnhancedSharedInput(h+β·z) → GatedCrossScaleSharedExpert
       → z_shared → refine → h_shared → PredHead
```

V2.4 的 Shared-Only 不再是传统 Shared-Only —— ExpertEnhancedSharedInput(hybrid) 将 expert 特征注入 shared 分支，且 V2.4 中 Shared-Only 首次**实际计算**了 expert 特征（V2.3 中 use_routed_branch=false 时不计算 expert）。

#### 发现 3: AdaptiveBranchGate 完全坍缩

```
Full Model (回滚后, ep120):
  branch_gate_shared_mean = 1.000000
  branch_gate_route_mean  = 0.000000 (2.04e-10)
  route_gamma = 0.018     (sigmoid(-4) ≈ 0.018)
  effective_route_ratio = 1.24   (h_route_proj 比 h_shared 幅度大 24%)
```

AdaptiveBranchGate 将所有样本的分支权重推向 shared=1.0, route=0.0，完全丧失自适应能力。`effective_route_ratio=1.24` 说明 route branch 产生了有意义且幅度更大的特征，但 gate 选择完全忽略它们。

这与 `shared_bias` 初始化一致（初始 softmax([1,-1])=[0.88,0.12]），但 gate 进一步将 route 压到 0。与 V2.3 的 `h_main = h_shared + 0.018·h_route_proj`（2% route）相比，V2.4 的 0% route 进一步削弱了 route branch 的贡献。

#### 发现 4: shared_input_beta 学习到的规律

```
            β_f     β_m     β_c
Full        0.365   0.597   0.337     mid 尺度需要最多 expert 增强
Shared-Only 0.393   0.548   0.201     mid 尺度需要最多 expert 增强
No Router   0.393   0.629   0.425     所有 beta 都很高
```

规律一致：mid 尺度（32→16，信息损失大）的 β 始终最高（0.55-0.63），说明 mid 尺度的 embedding 特征不足以支撑高质量重建，需要 expert 特征补偿。但过高的 β（0.3-0.6）意味着 shared 分支的输入已经被 expert 特征严重"污染"。

## 四、跨数据集洞察

### 4.1 TaxiBJ vs BikeNYC：为什么表现截然不同？

| 维度 | TaxiBJ | BikeNYC |
|------|--------|---------|
| 数据 MAE 量级 | ~12（大值，难预测） | ~3（小值，易预测） |
| embedding 质量 | 较弱（mid β=0.60） | 较强（mid β=0.21） |
| expert 特征贡献 | 高（依赖 expert） | 低（embedding 已足够） |
| shared branch 敏感性 | 极高（V2.4 改动致命） | 低（V2.4 改动无影响） |
| 最优架构 | Routed-Only（V2.3） | Full ≈ Shared-Only |

ExpertEnhancedSharedInput 对 TaxiBJ 的破坏性影响揭示了：**当 embedding 特征质量不足时，将 expert 特征混入 shared branch 可能并不是一个好策略**——expert 特征虽然增强了表示能力，但也引入了路由噪声，破坏了 shared branch 原有的稳定性。

### 4.2 AdaptiveBranchGate 的实际行为

两个数据集上，AdaptiveBranchGate 均未展现样本级自适应能力：

- **TaxiBJ**: 坍缩到 [1.0, 0.0]，route 完全被忽略
- **BikeNYC**: 收敛到 [0.975, 0.025]，接近 V2.3 的 residual 行为

输入信号 `[pool(h_shared)||pool(h_route)||q_f||scale_gate]` 的样本间差异不足以让 MLP 产生有意义的动态门控。门控学的本质是「全局偏好」而非「样本级自适应」。

## 五、结论与建议

### 5.1 核心结论

1. **超参数回滚有效但不充分**：LR/scheduler/lambda 回滚恢复了不使用 V2.4 组件的模型（Fine-Only 完全恢复），但使用 ExpertEnhancedSharedInput 的模型仍未恢复。

2. **ExpertEnhancedSharedInput(hybrid) 是 TaxiBJ 性能退化的架构根因**：将 expert 特征混入 shared 输入 (h+β·z) 对 TaxiBJ 的 shared branch 产生了负面影响。Shared-Only 从 14.82 升至 19.00（+28%），Full 从 12.01 升至 19.44（+62%）。

3. **BikeNYC 不受影响**：低 MAE + 高质量 embedding，expert 增强并非必需。

4. **AdaptiveBranchGate 未达预期**：样本级自适应未生效，退化为固定全局偏好。

### 5.2 建议

#### 短期（修复 TaxiBJ）

**方案 A: 将 TaxiBJ 的 shared_input_mode 改为 "pre"**
```json
// 保留 adaptive_gate 等 V2.4 架构，但 shared branch 退回到纯 embedding 输入
"shared_input_mode": "pre"
```
这样可以隔离问题：如果 shared_input_mode="pre" 恢复 V2.3 水平，则确认 ExpertEnhancedSharedInput 是唯一根因。

**方案 B: 将 branch_fusion_mode 改为 "residual"**
```json
// 退回到 V2.3 的分支融合方式
"branch_fusion_mode": "residual",
"route_gamma_init": -4.0
```
结合 shared_input_mode="pre"，应能完全恢复 V2.3 性能。

**方案 C: 降低 shared_input beta_init**
```json
// 减少 expert 特征对 shared 分支的初始影响
"shared_expert_beta_init": 0.01  // 当前 0.1
```

#### 中期（改进 V2.4 架构）

1. **Fix ExpertEnhancedSharedInput**: 考虑对 expert 特征做 stop_gradient (detach_shared_expert_input=true)，让 shared branch 仅使用 expert 的表示而不反向传播梯度干扰 expert 训练。

2. **Replace AdaptiveBranchGate**: 当前的样本级 gate 无实际作用。考虑：
   - Patch 级 gate（每个空间位置独立门控）
   - 或直接退回到 learnable scalar（residual 模式），去掉多余的 MLP 参数

3. **Ablation: shared_input_mode 对比**: 在 TaxiBJ 上跑 pre/post/hybrid/concat_hz 四种模式，量化每种模式的影响。

### 5.3 配置建议

TaxiBJ 当前最优配置组合（保守方案）：
