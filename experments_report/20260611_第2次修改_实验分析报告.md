## 1. 模型架构（基于代码实际实现）

### 1.1 整体架构

```
DualBranchSTImputer
├── Main Branch (MultiScaleMoEBackbone)
│   ├── [1] Multi-Scale Token Embedding (ScaleTokenEncoder ×3)
│   │   ├── embed_f: value_embed(Conv→GN→GELU) + mask_embed + scale_embed + time_embed + space_embed
│   │   ├── embed_m: 同上, H/2 × W/2
│   │   └── embed_c: 同上, H/4 × W/4
│   │
│   ├── [2] Quality-Aware Router (QualityRouter ×3)
│   │   ├── router_f, router_m, router_c
│   │   ├── 输入: pooled(h) || observation_stats(mask) || scale_embed
│   │   ├── observation_stats: [missing_rate, observed_ratio, temporal_missing_score,
│   │   │                        spatial_missing_score, aggregation_reliability]
│   │   └── 输出: softmax → expert selection probabilities [B, num_experts]
│   │
│   ├── [3] Intra-Scale Routed Expert Pool (TopKRoutedExpertPool, 参数共享)
│   │   ├── num_experts=4, top_k=2
│   │   ├── 每个 Expert: Conv3d→GN→GELU→ResidualSTBlock
│   │   ├── Routing: topk (硬选择) / soft_topk / dense
│   │   ├── 输出: Z_f [B,D,T,H,W], Z_m [B,D,T,H/2,W/2], Z_c [B,D,T,H/4,W/4]
│   │   └── share_experts=True: 三个尺度共用同一组专家
│   │
│   ├── [4] Cross-Scale Shared Expert (CrossScaleSharedExpert)
│   │   ├── 输入: h_f, h_m↑(upsample), h_c↑(upsample)
│   │   ├── Concat(h_f, h_m↑, h_c↑) → Conv1×1(dim*3→dim) → 2×ResidualSTBlock
│   │   └── 输出: Z_shared [B,D,T,H,W]
│   │
│   ├── [5] Progressive Route Fusion (ProgressiveRouteFusion)
│   │   ├── Stage 1: Coarse → Mid
│   │   │   ├── LearnableUpsample3D(Z_c): trilinear upsample → Conv3d→GN→GELU→ResidualSTBlock
│   │   │   └── GatedFusion2(Z_m, Z_c↑): Conv1×1(2D→2)→Softmax(dim=1)→加权和→ResidualSTBlock
│   │   │       → Z_mc, gate_16
│   │   └── Stage 2: Mid → Fine
│   │       ├── LearnableUpsample3D(Z_mc)
│   │       └── GatedFusion2(Z_f, Z_mc↑) → h_route, gate_32_route
│   │
│   ├── [6] Shared-Routed Residual Fusion (SharedRoutedResidualFusion)
│   │   ├── shared_refine: Z_shared → 2×ResidualSTBlock → h_shared
│   │   ├── route_proj: h_route → Conv1×1 → ResidualSTBlock → h_route_proj
│   │   ├── route_gamma: 可学习参数 (init=-4.0, σ(-4)≈0.018)
│   │   └── h_main = h_shared + sigmoid(route_gamma) × h_route_proj
│   │
│   └── [7] Prediction Head
│       └── Conv3d(D→D/2,k=3)→GELU→Conv3d(D/2→C_in,k=1) → x_hat_main
│
└── Aux Branch (NullResidualBranch, 当前禁用, alpha=0)
```

### 1.2 关键架构特点

| 特性 | 实现 | 说明 |
|---|---|---|
| **双分支解耦** | Routed Branch + Shared Branch 独立计算 | 路由分支做跨尺度渐进融合，共享分支做跨尺度特征交互 |
| **残差融合** | `h_main = h_shared + γ × h_route_proj` | 路由分支作为 shared 分支的残差修正（γ_init=-4 使初始贡献≈0） |
| **逐级门控** | Coarse→Mid→Fine 层级递进 | 非并行 concat，而是 coarse→mid→fine 逐步 upsample + gated fusion |
| **学习中上采样** | LearnableUpsample3D | trilinear upsample + Conv3D + GN + GELU + ResidualSTBlock 精炼 |
| **门控可解释** | gate_16 (coarse/mid), gate_32_route (fine/mid+coarse) | 可观察各尺度对最终预测的贡献 |
| **质量感知路由** | QualityRouter ×3 | 根据观测统计量动态调整专家选择 |
| **专家参数共享** | share_experts=True | 三尺度共享 4 个专家（共4个而非12个），减少参数量并促进跨尺度知识迁移 |

### 1.3 与旧版架构的核心差异

| 组件 | 旧版 | 新版（当前代码） |
|---|---|---|
| 路由分支融合 | 简单 `concat(Z_f, Z_m↑, Z_c↑)` → Conv1×1 | `ProgressiveRouteFusion`: Coarse→Mid→Fine 逐级门控 |
| 双分支融合 | `GatedFusion3(Z_f, Z_mc↑, Z_shared)` 三路融合 | `SharedRoutedResidualFusion`: shared 为主 + routed 残差修正 |
| 上采样 | 直接 `F.interpolate` | `LearnableUpsample3D` (upsample + 精炼) |
| 路由权重 | 显式参与 fusion gate | 通过可学习 γ (sigmoid) 控制贡献度 |
| 对应设计文档 | design_v2-2.md | design_v2-2(MOE融合模块改进).md |

## 2. 实验设计

### 2.1 消融实验矩阵

| # | 实验 | multiscale | router | share_exp | routed_br | shared_br | 等价描述 |
|---|------|:---:|:---:|:---:|:---:|:---:|---|
| 1 | **Fine-Only** | ❌ | ✅ | ✅ | ✅ | ❌ | 仅细尺度路由专家 |
| 2 | **No Router** | ✅ | ❌ | ✅ | ✅ | ✅ | 均匀门控, dense routing |
| 3 | **Fixed Experts** | ✅ | ❌ | ❌ | ✅ | ❌ | 每尺度独立专家池, dense routing |
| 4 | **No Cross-Scale** | ✅ | ✅ | ✅ | ✅ | ❌ | 无共享分支 = 纯路由分支 |
| 5 | **Routed-Only** | ✅ | ✅ | ✅ | ✅ | ❌ | 同上（结构等价于 No Cross-Scale） |
| 6 | **Shared-Only** | ✅ | ✅ | ✅ | ❌ | ✅ | 仅跨尺度共享专家 |
| - | **Full Model** | ✅ | ✅ | ✅ | ✅ | ✅ | 完整双分支 + 残差融合 |

> **注**: No Cross-Scale 与 Routed-Only 在结构上等价（均禁用 shared_br），差异仅来自训练随机性。后续分析中将两者合并讨论。

### 2.2 训练配置

| 参数 | TaxiBJ | BikeNYC |
|---|---|---|
| 空间分辨率 | 32×32 | 24×12 (padded) |
| 通道数 (c_in) | 2 | 2 |
| Full/Ablation Epochs | 120 / 100 | 100 / 100 |
| LR schedule | cosine: 1e-3 → 1e-6 | cosine: 1e-3 → 1e-6 |
| 缺失模式 | mixed, 40% | mixed, 40% |
| dim / experts / top_k | 64 / 4 / 2 | 64 / 4 / 2 |
| Loss | SmoothL1 + λ_cross=0.1 + λ_balance=0.01 | SmoothL1 + λ_cross=0.1 + λ_balance=0.01 |

### 2.3 Loss 函数

$$\mathcal{L} = \mathcal{L}_{\text{main}} + \lambda_{\text{cross}}\mathcal{L}_{\text{cross}} + \lambda_{\text{imp}}\mathcal{L}_{\text{importance}} + \lambda_{\text{load}}\mathcal{L}_{\text{load}} + \lambda_{\text{ent}}\mathcal{L}_{\text{fusion\_entropy}}$$

- $\mathcal{L}_{\text{main}}$: Masked SmoothL1, 缺失位置（fine scale）
- $\mathcal{L}_{\text{cross}}$: Cross-scale observed loss（mid + coarse 的观测位置一致性）
- $\mathcal{L}_{\text{importance}}$: 专家重要性均衡（各专家被选中的概率应均匀）
- $\mathcal{L}_{\text{load}}$: 专家负载均衡（各专家实际处理的样本数应均匀）
- $\mathcal{L}_{\text{fusion\_entropy}}$: 当前 λ=0，未启用

### 3.3 关键发现

1. **Full ≈ Shared-Only**: Full Model 仅比 Shared-Only 优 0.67%。路由分支的残差修正效果有限，BikeNYC 上共享分支是绝对主力。
2. **Routed-Only 表现良好**: MAE=2.89，距离 Full 仅差 2.1%，说明即使没有 shared branch，逐级门控融合也能较好地工作。
3. **Fine-Only 最差**: 单尺度 Fine-Only (MAE=3.07) 比 Full 差 8.5%，证明多尺度信息对 BikeNYC 至关重要。
4. **No Router 中等**: 去掉 QualityRouter（改用 uniform+dense）后 MAE=2.96，下降 4.7%，说明质量感知路由仍有价值。
5. **Fixed Experts 最差**: 独立专家池 + dense routing (MAE=3.12) 表现最差，说明专家参数共享和 Top-K 稀疏路由是关键设计。

### 3.4 收敛行为

```
Full Model (0611):    ep0=7.5 → ep20=3.5 → ep40=3.0 → ep60=2.85 → ep70=2.83* → ep100=2.91
Shared-Only (0611):   ep0=6.9 → ep20=3.3 → ep40=2.95 → ep60=2.88 → ep94=2.85* → ep100=2.89
```

- Full Model 在 ep70 达到最佳，之后略微过拟合
- Shared-Only 收敛慢但持续改善到 ep94
- Fine-Only 快速收敛后 plateau，提升空间有限

### 4.3 跨版本退化分析（TaxiBJ Fine-Only）

```
Fine-Only 趋势: 12.09 (0609) → 12.41 (0610) → 12.78 (0611)
Full Model 趋势: 14.47 (0609) → 12.77 (0610) → 13.54 (0611)
Shared-Only 趋势: 13.19 (0601) → 12.91 (0610) → 13.92 (0611)
```

**退化原因分析**:
1. **新 loss 项副作用**: λ_importance_balance=0.01 和 λ_load_balance=0.01 对简单配置（Fine-Only、Shared-Only）产生不必要约束
2. **Fine-Only 不含多尺度**: Fine-Only 只有细尺度 → 没有跨尺度 load balance 需求 → 但 loss 仍然计算并施加了均衡约束
3. **权重竞争**: λ_cross=0.1 对 Fine-Only 可能过大（无多尺度时 cross-scale loss 退化为自身一致性约束）

### 4.4 关键发现

1. **Fine-Only 三连冠**: 在 3 个批次中 Fine-Only 均为 TaxiBJ 最佳，但有明显退化趋势
2. **Full Model 不敌 Fine-Only**: 多尺度 + 双分支架构在 TaxiBJ 上没有带来增益，反增噪声
3. **Routed-Only 崩溃**: MAE=19.72，远差于 Shared-Only (13.92)，说明 TaxiBJ 上路由分支单独工作不稳定
4. **No Router 大幅退步**: 旧代码(0529)时为 12.49，新代码 17.95，新 fusion 依赖 quality routing
5. **Fixed Experts 中等**: MAE=15.34，独立专家池 + dense routing 在 TaxiBJ 上有一定合理性

### 4.5 收敛行为

```
Full Model (0611):    ep1=48 → ep20=25 → ep40=19 → ep60=16 → ep80=14.2 → ep94=13.5* → ep100=14.1 → ep120=14.1
Fine-Only (0611):     ep1=53 → ep20=22 → ep40=15 → ep60=13.4 → ep86=12.78* → ep100=12.88
Shared-Only (0611):   ep1=55 → ep20=26 → ep40=18 → ep60=15.5 → ep90=13.92* → ep100=14.16
No Router (0611):     ep1=55 → ep20=28 → ep40=22 → ep61=17.95* → ep100=18.46
```

- Full Model 在 ep94 达最佳后过拟合（cosine lr 已极低但 loss 反弹）
- Fine-Only 收敛最快且最稳定
- No Router 收敛快但 plateau 高

### 5.1 排名对比

| 排名 | BikeNYC | MAE | TaxiBJ | MAE |
|:---:|---|---:|---|--:|
| 🥇 | **Full Model** | 2.83 | **Fine-Only** | 12.78 |
| 🥈 | Shared-Only | 2.85 | Full Model | 13.54 |
| 🥉 | Routed-Only | 2.89 | Shared-Only | 13.92 |
| 4 | No Router | 2.96 | Fixed Experts | 15.34 |

### 5.2 架构组件贡献度（ΔMAE vs Full）

| 消融的组件 | BikeNYC 损失 | TaxiBJ 损失 | 含义 |
|---|---|---|---|
| 路由分支 (Shared-Only) | +0.67% | +2.87% | TaxiBJ 更需要路由分支 |
| 共享分支 (Routed-Only) | +2.10% | **+45.71%** | TaxiBJ 极其依赖共享分支 |
| Quality Router (No Router) | +4.69% | +32.59% | TaxiBJ 对质量感知路由更敏感 |
| 多尺度 (Fine-Only) | +8.55% | **-5.94%** | TaxiBJ 多尺度反而有害！ |
| 专家共享 (Fixed Experts) | +10.28% | +13.30% | 两数据集都受益于专家共享 |

### 5.3 核心矛盾

```
BikeNYC: 多尺度 + 共享分支 = 最佳  (Full Model)
TaxiBJ:  单尺度 + 路由分支 = 最佳  (Fine-Only)
```

**可能原因**:
1. **空间分辨率差异**: TaxiBJ 32×32 → 降采样到 16×16/8×8 后信息损失大；BikeNYC 24×12 → 12×6/6×3，降采样比例相同但原始分辨率低，mid/coarse 仍能保留足够信息
2. **空间结构差异**: TaxiBJ 是规则网格（北京出租车轨迹），空间平滑度高 → 细尺度已足够；BikeNYC 是路网分布，需要多尺度捕获不同粒度的空间模式
3. **数据复杂度**: TaxiBJ 流量模式更复杂（32×32 网格，多中心），多尺度混入噪声；BikeNYC 结构简单，多尺度带来互补信息

### 5.4 各批次稳定性

```
BikeNYC (跨批次标准差/均值):
  Full: 2.81-2.95, σ/μ≈2.1%  — 非常稳定
  Shared-Only: 2.82-2.90, σ/μ≈2.3%
  Fine-Only: 3.06-3.16, σ/μ≈1.5%

TaxiBJ (跨批次标准差/均值):
  Fine-Only: 12.09-12.78, σ/μ≈2.4%
  Full Model: 12.77-14.47, σ/μ≈5.3%  — 非常不稳定
  Shared-Only: 12.91-13.92, σ/μ≈3.2%
  No Router: 12.49-20.59, σ/μ≈19%  — 极其不稳定
```

BikeNYC 所有配置都非常稳定。TaxiBJ 的 Full Model 和 No Router 跨批次波动极大，说明训练对初始化和超参数敏感。

## 7. 综合评估与建议

### 7.1 架构有效性

| 设计组件 | 有效性 | 证据 |
|---|---|---|
| ProgressiveRouteFusion (逐级门控) | ✅ 有效 | Routed-Only 在 BikeNYC 上 MAE=2.89，说明独立路由分支工作良好 |
| SharedRoutedResidualFusion (残差融合) | ⚠️ 部分有效 | Full vs Shared-Only 提升微弱 (BikeNYC: 0.7%, TaxiBJ: 2.9%)，路由残差贡献有限 |
| CrossScaleSharedExpert | ✅ 关键 | Shared-Only 在两数据集上都排前3 |
| QualityRouter | ✅ 有效 | No Router 退步显著，尤其 TaxiBJ |
| TopKRoutedExpertPool (top-k) | ✅ 有效 | 对比 Fixed Experts 有明显提升 |
| route_gamma_init (-4.0) | ⚠️ 可能过于保守 | γ≈0.018 意味着路由分支几乎不参与初始训练，可能训练结束时 γ 仍然很小 |

### 7.2 数据集特异性建议

**BikeNYC**:
- 当前 Full Model 是最优选择
- 可考虑增大 route_gamma_init（如 -2.0 → σ≈0.12），让路由分支更早参与训练
- 100 epochs 充足，可尝试 80 epochs 早停

**TaxiBJ**:
- **优先使用 Fine-Only**，避免多尺度引入噪声
- 若必须使用 Full Model，需：
  - 降低 λ_cross（多尺度 loss 对 32×32 可能过约束）
  - 对非多尺度配置关闭 balance loss
  - 增大 route_gamma_init 让残差融合更活跃
- 120 epochs 可减至 100 epochs（Fine-Only 在 ep86 即达最佳）

### 7.3 下一步方向

1. **TaxiBJ Fine-Only 退化修复**: 对无多尺度的配置禁用 cross-scale loss 和 load balance loss
2. **route_gamma 调参**: 尝试 -2.0 或 -1.0，让路由残差更早激活
3. **TaxiBJ 特定架构**: 考虑减少 mid/coarse 的降采样比例（如 fine_to_mid=2→1, fine_to_coarse=4→2）
4. **Aux Branch 启用实验**: 当前 alpha=0，可尝试启用 aux branch 观察效果
5. **Fusion Entropy Loss**: 尝试启用 λ_fusion_entropy > 0，促进 gate_16/gate_32 更确定的门控决策
6. **Baselines 对比**: 与统计方法（mean/historical/linear）和经典深度方法（SAITS/Transformer/Conv3D-UNet）对比
