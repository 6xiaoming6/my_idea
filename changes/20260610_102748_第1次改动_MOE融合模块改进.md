# MOE融合模块改进

## 修改时间

2026-06-10 10:27:48

## 修改背景

本次修改根据 `design_v2-2(MOE融合模块改进).md` 实现主分支最后阶段的多尺度融合升级。

修改前，主分支从三个尺度专家和共享专家得到特征后，采用较简单的融合方式：

```text
z_f:      [B, D, T, H,   W  ]
z_m:      [B, D, T, H/2, W/2] -> trilinear upsample -> [B, D, T, H, W]
z_c:      [B, D, T, H/4, W/4] -> trilinear upsample -> [B, D, T, H, W]
z_shared: [B, D, T, H,   W  ]

concat([z_f, z_m_up, z_c_up, z_shared]) -> Conv3d(dim * 4, dim) -> ResBlock -> h_main
```

这种方式可以跑通，但粗尺度 `z_c` 会直接从 `H/4 x W/4` 跳到 `H x W`，没有显式建模 coarse -> mid -> fine 的层级关系，也没有显式判断每个位置应该更依赖哪个尺度。

## 修改文件

- `src/stmoe_imputer/models/fusion.py`
- `src/stmoe_imputer/models/main_branch.py`
- `src/stmoe_imputer/models/__init__.py`

## 代码层面的改动

### 1. 新增融合模块文件

新增 `src/stmoe_imputer/models/fusion.py`，集中实现多尺度融合相关模块，避免把主分支文件写得过重。

新增类如下：

- `LearnableUpsample3D`
  - 先使用 `F.interpolate(..., mode="trilinear")` 对齐时空尺寸。
  - 再接 `Conv3d -> GroupNorm -> GELU -> ResidualSTBlock` 做可学习细化。
  - 作用是替代单纯插值，让低分辨率尺度在升到高分辨率后可以学习局部修正。

- `GatedFusion2`
  - 输入两个同尺寸特征，例如 `z_m` 和 `z_c_to_m`。
  - 将两个特征在通道维拼接后，用 `Conv3d(dim * 2, 2, kernel_size=1)` 生成两个分支的门控 logits。
  - 对分支维做 `softmax(dim=1)`，得到每个时空位置的两个权重。
  - 输出 `gate[:, 0:1] * x1 + gate[:, 1:2] * x2`，再经过 `ResidualSTBlock` refinement。

- `GatedFusion3`
  - 输入三个同尺寸特征，例如 `z_f`、`z_mc_to_f`、`z_shared`。
  - 使用 `Conv3d(dim * 3, 3, kernel_size=1)` 生成三路门控权重。
  - 对 fine、中粗融合、shared 三个来源做逐位置加权融合。

- `ProgressiveScaleGatedFusion`
  - 串联 `LearnableUpsample3D`、`GatedFusion2`、`LearnableUpsample3D`、`GatedFusion3`。
  - 返回最终 `h_main`，同时返回中间特征和门控图，方便后续可视化分析。

### 2. 替换主分支旧融合逻辑

在 `src/stmoe_imputer/models/main_branch.py` 中：

- 删除原来的 `self.fusion = nn.Sequential(...)`，也就是 `Conv3d(dim * 4, dim) + ResidualSTBlock + ResidualSTBlock` 这一路简单融合。
- 新增 `self.progressive_fusion = ProgressiveScaleGatedFusion(...)`。
- 删除 forward 中 `z_m_up = F.interpolate(...)`、`z_c_up = F.interpolate(...)` 和 `torch.cat([z_f, z_m_up, z_c_up, z_shared], dim=1)` 的简单拼接逻辑。
- 改为调用：

```python
fusion_outputs = self.progressive_fusion(
    z_f=z_f,
    z_m=z_m,
    z_c=z_c,
    z_shared=z_shared,
)
h_main = fusion_outputs["h_main"]
x_hat_main = self.pred_head(h_main)
```

### 3. 保留消融配置兼容性

原模型支持 `use_multiscale`、`use_shared_branch`、`use_routed_branch` 等配置。本次修改保留这些配置的基本语义：

- `use_routed_branch=False` 时，`z_f/z_m/z_c` 仍会置零。
- `use_shared_branch=False` 或 `use_multiscale=False` 时，`z_shared` 仍会置零。
- `use_multiscale=False` 时，`z_m` 和 `z_c` 也会置零，再进入新的 progressive fusion。

这样旧的消融实验配置仍能运行，只是最后融合模块统一经过新的逐级门控结构。

### 4. 增加可解释性输出

在 `outputs["gates"]` 中新增：

- `fusion_16`: `[B, 2, T, H/2, W/2]`
  - 第 0 个通道表示 mid 特征权重。
  - 第 1 个通道表示 coarse 上采样到 mid 后的权重。

- `fusion_32`: `[B, 3, T, H, W]`
  - 第 0 个通道表示 fine 特征权重。
  - 第 1 个通道表示 mid-coarse 融合特征权重。
  - 第 2 个通道表示 shared cross-scale 特征权重。

在 `outputs["features"]` 中新增：

- `z_c_to_m`: coarse 特征可学习上采样到 mid 尺度后的结果。
- `z_mc`: mid 与 coarse-to-mid 门控融合后的中粗尺度特征。
- `z_mc_to_f`: `z_mc` 可学习上采样到 fine 尺度后的结果。

这些输出可以用于后续画门控热力图，分析不同缺失模式下模型依赖哪个尺度。

### 5. 更新包导出

在 `src/stmoe_imputer/models/__init__.py` 中导出：

- `LearnableUpsample3D`
- `GatedFusion2`
- `GatedFusion3`
- `ProgressiveScaleGatedFusion`

后续如果要单独写消融实验、单元测试或可视化脚本，可以直接从 `stmoe_imputer.models` 导入这些模块。

## 模型层面的改动

### 修改前的主分支融合架构

修改前的主分支整体流程可以概括为：

```text
x_f, m_f -> fine embedding  -> router_f -> routed experts -> z_f
x_m, m_m -> mid embedding   -> router_m -> routed experts -> z_m
x_c, m_c -> coarse embedding -> router_c -> routed experts -> z_c

h_f, h_m, h_c -> CrossScaleSharedExpert -> z_shared

z_m -> upsample to fine -> z_m_up
z_c -> upsample to fine -> z_c_up

concat(z_f, z_m_up, z_c_up, z_shared)
  -> Conv3d(dim * 4, dim)
  -> ResidualSTBlock
  -> ResidualSTBlock
  -> h_main
  -> pred_head
  -> x_hat_main
```

这个结构把 fine、mid、coarse、shared 四路信息一次性拼接到 fine 尺度，没有显式体现尺度层级。

### 修改后的主分支融合架构

修改后的主分支在专家和共享专家之前的部分不变：

```text
x_f, m_f -> fine embedding  -> router_f -> routed experts -> z_f
x_m, m_m -> mid embedding   -> router_m -> routed experts -> z_m
x_c, m_c -> coarse embedding -> router_c -> routed experts -> z_c

h_f, h_m, h_c -> CrossScaleSharedExpert -> z_shared
```

变化集中在最后融合阶段。新的融合流程为：

```text
Stage 1: coarse -> mid

z_c: [B, D, T, H/4, W/4]
  -> LearnableUpsample3D
  -> z_c_to_m: [B, D, T, H/2, W/2]

z_m:      [B, D, T, H/2, W/2]
z_c_to_m: [B, D, T, H/2, W/2]
  -> GatedFusion2
  -> z_mc: [B, D, T, H/2, W/2]


Stage 2: mid -> fine

z_mc: [B, D, T, H/2, W/2]
  -> LearnableUpsample3D
  -> z_mc_to_f: [B, D, T, H, W]

z_f:       [B, D, T, H, W]
z_mc_to_f: [B, D, T, H, W]
z_shared:  [B, D, T, H, W]
  -> GatedFusion3
  -> h_main: [B, D, T, H, W]

h_main -> pred_head -> x_hat_main: [B, C, T, H, W]
```

以 TaxiBJ 常用的 `H=W=32` 为例，形状是：

```text
z_c:       [B, D, T, 8,  8 ]
z_c_to_m:  [B, D, T, 16, 16]
z_m:       [B, D, T, 16, 16]
z_mc:      [B, D, T, 16, 16]
z_mc_to_f: [B, D, T, 32, 32]
z_f:       [B, D, T, 32, 32]
z_shared:  [B, D, T, 32, 32]
h_main:    [B, D, T, 32, 32]
x_hat_main:[B, C, T, 32, 32]
```

## 改动后的模型含义

新的融合模块让主分支最后的多尺度融合从“简单拼接”变成“逐级尺度对齐 + 动态门控选择”。

直观理解：

- coarse 特征负责更大范围的趋势和全局上下文。
- mid 特征负责区域级结构。
- fine 特征负责局部细节。
- shared 特征负责跨尺度共享信息。

模型现在会先把 coarse 信息注入 mid，形成 `z_mc`，再把 `z_mc` 注入 fine，并和 `z_f`、`z_shared` 做动态融合。门控权重是按时空位置生成的，因此不同时间、不同空间网格可以选择不同的信息来源。

这对缺失补全任务更合理：

- 小范围随机缺失时，fine 权重可能更高。
- 连续空间块缺失时，mid-coarse 权重可能更高。
- 复杂或大范围缺失时，shared cross-scale 权重可能更高。

## 验证结果

已使用以下命令完成 smoke 测试：

```bash
conda run -n difftdi python scripts/smoke_test.py
```

测试通过，关键输出如下：

```text
x_hat_main: (2, 2, 6, 16, 16)
h_st_aux:   (2, 16, 6, 16, 16)
fusion_16:  (2, 2, 6, 8, 8)
fusion_32:  (2, 3, 6, 16, 16)
```

smoke 配置中 fine 尺度是 `16 x 16`，所以 `fusion_16` 对应 mid 尺度 `8 x 8`，`fusion_32` 对应 fine 尺度 `16 x 16`。如果使用 TaxiBJ 的 `32 x 32` 配置，则预期为：

```text
fusion_16: [B, 2, T, 16, 16]
fusion_32: [B, 3, T, 32, 32]
```

loss 正常计算，说明新融合模块的 forward、输出字典和现有 loss 逻辑可以衔接。

## 注意事项

- 当前 balance loss 仍只统计专家路由门控 `fine/mid/coarse`，没有把 `fusion_16/fusion_32` 纳入正则项。
- 本次没有引入 reliability map，文档中提到的 mask/reliability 融合可以作为后续增强点。
- 原来的调试特征 `z_m_up`、`z_c_up` 被新的 `z_c_to_m`、`z_mc`、`z_mc_to_f` 替代；当前仓库没有发现其他脚本依赖旧特征名。
